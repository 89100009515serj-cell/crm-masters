# CRM Мастеров

Внутренняя CRM для генерации и обработки заявок мастеров. Next.js 15 (App Router) + TypeScript + Tailwind + Prisma/PostgreSQL + NextAuth v5.

## Запуск

```bash
npm install
cp .env.example .env   # заполнить DATABASE_URL и AUTH_SECRET
npx prisma migrate dev --name init
npm run seed            # создаст admin@crm.local / admin123
npm run dev
```

## Что готово (Этап 1–8)

Этапы 1–7 — см. выше. Этап 8 — **Архив + Поиск**:

- `src/server-actions/search-requests.ts` — единая точка поиска/фильтрации/пагинации.
  `scope: "active" | "archive"` определяет набор статусов по умолчанию (`ACTIVE_STATUSES` /
  `ARCHIVE_STATUSES`), поиск по `query` ищет через Prisma `OR` сразу по номеру заявки (id),
  ФИО клиента, телефону, адресу, описанию работ и имени мастера (`insensitive`-сравнение)
- `/requests` теперь показывает только активные заявки (NEW/ASSIGNED/ACCEPTED/ON_SITE)
- `/archive` — новая страница: COMPLETED + CANCELLED, таблица в режиме `viewOnly` (без кнопки
  «Перенести»), та же логика поиска/фильтров/сортировки/пагинации
- Пагинация по 20 заявок на страницу (компонент `Pagination`), фильтры/поиск/сортировка/номер
  страницы хранятся в query-параметрах URL — страницу можно перезагрузить или сохранить в
  закладки с тем же состоянием
- Индексы в Prisma-схеме на `clientName`, `clientPhone`, `clientAddress` под поиск
- Ссылка «Архив» в навигации (`AppHeader`) для всех ролей с доступом к заявкам
- `/api/requests/export` поддерживает `scope` и `q`, экспортирует ровно то, что видно в
  таблице на момент клика

### Ограничение поиска по номеру заявки

Заявки используют `cuid` (например `cm4x7k9...`) — это длинный технический ID без понятного
человеку "номера". Поиск ищет вхождение текста прямо в этот ID. Если нужен короткий
читаемый номер (как в Telegram-уведомлениях, последние 6 символов), стоит добавить
автоинкрементное поле `displayNumber Int @unique` с триггером/sequence — это отдельная
небольшая доработка, сейчас не сделана, чтобы не плодить дублирующиеся идентификаторы.

Этапы 1–6 — см. ниже. Этап 7 — **Telegram-уведомления**:

- `src/lib/telegram.ts` — отправка через Bot API напрямую (`fetch`, без сторонних SDK),
  HTML-шаблоны сообщений под каждое событие, неблокирующая постановка в очередь
  (`enqueue`, не задерживает ответ server action; интерфейс можно заменить на
  настоящую очередь типа BullMQ/QStash при росте нагрузки)
- Поля `User.telegramChatId`, `telegramEnabled`, `telegramNotifyTypes`,
  `Notification.deliveredToTelegram` — в Prisma-схеме
- `/profile` — страница подключения Telegram (ввод chat_id → тестовое сообщение → сохранение),
  переключатель вкл/выкл, для ADMIN/CURATOR — чекбоксы какие именно уведомления получать
- Интеграция в server actions: `createRequest`/`assignMaster` (назначение), `rescheduleRequest`
  (перенос, с причиной и новой датой), `acceptRequest` («Принял» → уведомление создателю),
  `arriveAtSite` («На адресе» → уведомление создателю, опционально для менеджера/куратора)
- `scripts/telegram-bot.ts` — отдельный long-polling процесс, отвечает на `/start` своим
  `chat_id`, чтобы пользователь мог его скопировать в профиль (см. ниже)

## Telegram-бот: создание и настройка

1. В Telegram напишите **@BotFather**, отправьте `/newbot`, придумайте имя и username.
   BotFather пришлёт токен вида `123456:ABC-DEF...` — это `TELEGRAM_BOT_TOKEN`.
2. Впишите токен в `.env`, а также `APP_URL` (адрес, на который ведут ссылки в сообщениях).
3. Запустите бота для выдачи ID отдельной командой:
   ```bash
   npm run telegram-bot
   ```
   Это самостоятельный long-polling процесс (не часть `next dev`/`next start`) — держите его
   запущенным постоянно (например, через `pm2` или systemd-юнит на проде).
4. Пользователь пишет боту `/start`, получает свой Telegram ID, вставляет его на `/profile`
   в CRM и нажимает «Подключить Telegram» — бот сразу пришлёт тестовое сообщение.

Если в будущем понадобятся интерактивные кнопки/команды (а не только выдача ID), этот же
скрипт — подходящее место для расширения через `telegraf`, не затрагивая Next.js-приложение.

- Prisma-схема: User/Role, ServiceType, Source, Request (+ `receiptPhotoUrls String[]`),
  RescheduleLog, Notification
- Auth, Permissions, финансовые расчёты на `Decimal` (40% / 4% / 5% / остаток)
- Server actions: заявки, перенос, мастера, фото чека, уведомления (см. предыдущие этапы)
- Загрузка фото чека (1–3 файла), просмотр у ADMIN/CURATOR
- Экспорт заявок в Excel с учётом фильтров (`/api/requests/export`)
- In-app уведомления (колокольчик, опрос раз в 30 сек)
- **Аналитика и дашборды по ролям** (`src/lib/analytics.ts`):
  - ADMIN/CURATOR: выручка, прибыль (только ADMIN), кол-во выполненных, средний чек/расход,
    топ-3 мастера по кол-ву и заработку, статистика по источникам (кол-во/сумма/конверсия),
    статистика по сервисам, график выручки по дням (Recharts LineChart), круговая диаграмма
    распределения по статусам (Recharts PieChart)
  - MANAGER: только свои созданные заявки — кол-во, конверсия, средний чек
  - MASTER: карточки — выполнено за период, заработок, средний чек, топ-3 сервиса
  - Выбор периода: Сегодня / 7 дней / 30 дней / Весь период / произвольный диапазон —
    управляется через query-параметры `/dashboard?period=...&from=...&to=...`
  - Тяжёлые запросы кэшируются через `unstable_cache` (60 сек, тег `analytics`)
  - Кнопка «Экспорт аналитики в Excel» (`/api/analytics/export`) — сводка + источники +
    сервисы + мастера на отдельных листах
- Страницы: `/login`, `/dashboard`, `/requests`, `/requests/new`, `/masters`, `/my-requests`
  (завершённые заявки автоматически уходят из активного списка мастера)
- Общий хедер с навигацией (`AppHeader`) на десктоп-страницах, колокольчик — везде

## Методология аналитики

Количество и конверсия заявок считаются по `createdAt` (когда заявка появилась), а выручка/
прибыль/средний чек — по `completedAt` (когда фактически прошли деньги). Это даёт корректную
выручку за период, даже если заявка создана раньше периода, но закрыта внутри него.

## Важно про хранение фото

Сейчас фото чеков пишутся на диск сервера (`/public/uploads/receipts`). Это работает на
обычном Node-сервере (VPS, Docker), но **не подойдёт для serverless-деплоя** (Vercel и т.п.),
где файловая система эфемерна. Перед таким деплоем замените запись в `uploadReceiptPhotos`
(`src/server-actions/receipt-photos.ts`) на загрузку в Supabase Storage / S3.
